### 性能优化

**LUT 加速（最重要，4倍提速）：**

发现 `params_schema.py` 里 `use_enlarger_lut` 和 `use_scanner_lut` 默认值是 `False`，但 `params_mapper.py` 里设为 `True`。`save_output_layer` 直接调 `digest_params` 不经过 `params_mapper`，导致保存时不走 LUT 加速。

修复：把默认值改为 `True`。

```python
# params_schema.py
use_enlarger_lut: bool = True   # 原来是 False
use_scanner_lut: bool = True    # 原来是 False
```

效果：全分辨率扫描从 **26 秒 → 4.5 秒**（3000×2000，无颗粒）。

**白平衡色度适应加速（13倍提速）：**

`_apply_white_balance_adaptation` 用 `colour-science` 全流程（RGB→XYZ→LMS→adapt→XYZ→RGB），对 3000×2000 图像耗时 1570ms。

Von Kries 色度适应本质是一个 3×3 矩阵乘法，三个矩阵可以预先合并。

```python
M_cat = colour.adaptation.matrix_chromatic_adaptation_VonKries(source_white, target_white)
combined = (M_to_rgb @ M_cat @ M_to_xyz).astype(np.float32)
result = (rgb.reshape(-1, 3) @ combined.T).reshape(h, w, 3)
```

效果：色度适应从 **1570ms → 50ms**，custom 白平衡总时间从 2.1 秒降到 0.6 秒（和 as_shot 持平）。

**预览分辨率：**
- 默认预览尺寸从 640px 提到 1200px（`params_schema.py`）
- 预览清晰度提升 3.5 倍，速度约 3 秒

## 2026-05-20（续）— 第三轮对话

### 白平衡功能深度审查（代码级）

**背景：** 上一轮讨论了白平衡的定位问题，这一轮读了代码，做了更彻底的分析。

---

#### 代码层面的发现

**`reference_illuminant` 是什么：**

每种胶片 profile 里有一个 `reference_illuminant` 字段（默认值 `D55`，即标准日光）。这是胶片出厂时厂家测量感光曲线时用的光源，写死在胶片数据文件里，是胶片的固有属性，不是用户能改的东西。

它的作用是告诉光谱上采样算法（hanatos2025/mallett2019）：这卷胶片的感光曲线是在什么光源下测出来的，计算时以此为基准。

**用户层白平衡控件做的事：**

`LoadRawSection` 里的白平衡（`as_shot` / `daylight` / `tungsten` / `custom`）作用在 RAW 解码阶段，在胶片模拟之前：

```
RAW 文件 → [白平衡适配] → 线性 ACES RGB → [胶片模拟 pipeline] → 输出
```

改变白平衡 = 改变送进胶片模拟的 RGB 数值的色温 = 整个 pipeline 重跑。

---

#### 这个软件处理的是数码照片

这个软件的本质是：把数码照片"假装"成胶片拍的。

```
数码照片（RAW 或 RGB）
    ↓
喂给虚拟胶片
    ↓
虚拟胶片按照真实胶片的化学特性"曝光"
    ↓
虚拟暗房"冲洗"
    ↓
输出带有胶片质感的图像
```

白平衡在这里的问题：数码照片进入"虚拟胶片"之前，它的颜色是什么，直接决定了胶片"看到"的是什么光。改变白平衡 = 改变输入给胶片模拟的原始数据 = 整个 5 秒 pipeline 重新跑。

---

#### 最终结论

**对专业摄影师，`as_shot` 已经是正确答案。**

专业摄影师拍 RAW 时，相机白平衡是有意为之的，或者在 Lightroom/Capture One 里已经调好才导出。`as_shot` = 用相机记录的白平衡系数解码，这是 RAW 工作流的标准做法。

`custom` 模式的实际价值极低：
- 场景是"拿到别人拍的 RAW，相机白平衡设错了"
- 但这个场景本来就应该在 Lightroom 里做，不是在胶片模拟工具里做
- 而且即使 RAW 解码优化到 0.6s，后续 pipeline 还要 4.5s，体验依然差

**白平衡控件对核心用户几乎没有价值，但带来了严重的速度感知问题。**

## 2026-05-21 — 第六轮对话

### 探索：导出预设给其他软件

**目标：** 导出一个文件，导入 Capture One，完整复原 spektrafilm 的颜色。

**探索过程：**
- 尝试导出 .cube LUT → 验证发现对 RAW 工作流无效（两个软件 RAW 解码起点不同）
- 尝试导出 ICC Profile + costyle → 拆解 Capture One 自带的 1461 个相机 profile，确认格式（scnr + Lab PCS + mft2），生成并验证 ICC 文件合法，但颜色完全错误
- 根本原因：Capture One 的 ICC 作用在 RAW 解码阶段，输入是相机传感器数据，与 spektrafilm 的输入（已解码图像）不同，无法精确复原
- Capture One Match Look 工具（AI 近似匹配）不是精确复原

**结论：** 在 RAW 工作流下，任何"导出预设给其他软件精确复原颜色"的方案都走不通。正确工作流是 spektrafilm 处理完保存 TIFF，在 Capture One 里直接用这张 TIFF。

**已移除：** 导出 LUT 按钮、导出 C1 Style 按钮及所有相关代码。

---

### 探索：GPU 实时预览

**目标：** 颗粒和 Halation 在 preview_mode 下被关闭，用户调参看不到效果，希望用 GPU 做近似实时预览。

**探索过程：**
- 验证环境：Apple M2 Max，Metal 4，wgpu 0.31.0 安装成功，GPU compute shader 验证通过（10ms 处理 1200×800）
- 搭建了 `gpu_preview/` 子模块框架（halation.py + pipeline.py），接入 controller.py
- 发现根本问题：GPU 路径是"CPU 跑完整 pipeline（关 Halation）+ GPU 叠加 Halation"，总时间 = CPU 时间 + 10ms，比纯 CPU 还慢
- 真正的加速需要把整个 pipeline（光谱上采样、密度曲线、卷积）全部用 WGSL shader 重写，这是月级别的工程
- 换 4090 也无法解决问题，瓶颈在 CPU 上的 Numba JIT 物理模拟，不在 GPU 算力

**结论：** GPU 实时预览在当前架构下不可行，需要完整重写 preview pipeline 为 GPU shader 近似版本，工程量过大，暂时搁置。

**已移除：** gpu-lab 目录（用户手动删除），架构文档第八节 GPU 预览内容，steering 文件中的 GPU 方向。

**给下一个智能体的提示：** 如果未来重新考虑 GPU 预览，正确路径是：不依赖 CPU pipeline，完全用 WGSL compute shader 重写一套近似渲染器（色彩变换用 3D LUT 采样，Halation 用高斯模糊，颗粒用 hash noise），绕开 Numba 的物理模拟。这是独立工程，不是对现有代码的加速。

### 重要讨论：白平衡功能审查

**结论：** 白平衡控件对专业摄影师几乎没有价值，但每次触发都要重跑 5 秒 pipeline。

- `reference_illuminant` 是胶片 profile 的出厂属性，不是用户控件
- 用户层白平衡作用在 RAW 解码阶段，改一次 = 整个 pipeline 重跑
- `as_shot` 对专业摄影师已经是正确答案
- 待执行：白平衡三个控件从主面板移到高级 Tab（已确认，未执行）

### 重要讨论：产品设计原则

**核心结论：** 用户永远按自己的心智模型使用产品，不按产品的专业度。

设计应对策略：
1. **期望管理**：不给用户一个看起来像"白平衡滑块"的东西然后让他等 5 秒
2. **功能分层**：第一眼看到的是"能立刻用、立刻看到结果"的东西
3. **反馈即时**：操作后必须立刻有反馈，沉默等待是最差体验
4. **工作流顺序**：先定构图曝光，再定色彩，最后加质感
